# Zeffy API integration

Free For Charity uses **Zeffy** (the free fundraising platform) for donations, events, and
ticketing. As of 2026 Zeffy ships a **public REST API**, so FFC can pull this data programmatically
instead of the manual CSV import that workflow 213 still supports. This doc is the reference for
that integration.

## API at a glance

- **Base URL:** `https://api.zeffy.com` — REST, JSON, OpenAPI 3.1.
- **Auth:** `Authorization: Bearer <API_KEY>` — an **organization API key** (JWT), generated by the
  **org owner** in Zeffy → Settings → Integrations. Free for every Zeffy org.
- **Read-only — no write capabilities.** Every operation in the OpenAPI spec is a `GET`. There are
  **no** `POST`/`PUT`/`PATCH`/`DELETE` endpoints, so you cannot create or modify campaigns,
  contacts, or payments through the API. The only inbound direction is Zeffy → you via the
  `payment.completed` webhook (data delivery, not a write API). Three resources, all `GET`:

  | Endpoint                     | Returns                                                                                                               |
  | ---------------------------- | --------------------------------------------------------------------------------------------------------------------- |
  | `GET /api/v1/payments`       | Donations/ticket payments. Filters: `status`, `type`, `currency`, `campaign`, `contact`, `created[gte\|gt\|lte\|lt]`. |
  | `GET /api/v1/payments/{id}`  | One payment with line items, refunds, dispute, recurring, receipt URL.                                                |
  | `GET /api/v1/contacts`       | Donors. Filters: `email`, `created[*]`, `updated[*]`. Lifetime giving stats + address.                                |
  | `GET /api/v1/contacts/{id}`  | One donor profile.                                                                                                    |
  | `GET /api/v1/campaigns`      | Donation forms + events. Filters: `created[*]`. Target vs. volume, occurrences.                                       |
  | `GET /api/v1/campaigns/{id}` | One campaign with occurrences (multi-date events).                                                                    |

- **Pagination:** cursor-based — each list response has `has_more` and `next_cursor`; pass
  `next_cursor` as `starting_after` (`limit` 1–100, default 10). The helper pages this
  automatically.
- **Rate limit:** 100 requests/minute per key (HTTP `429`); the helper backs off and retries.
- **Amounts:** in **cents**. Exports keep the raw cents column and add a dollars convenience column.
- **Timestamps:** Unix **seconds**. Exports convert to ISO 8601 UTC.
- **Webhook (push):** Zeffy can `POST` a `payment.completed` event (full payment object) to a URL
  you configure in the same Integrations dialog. Respond `2xx` to acknowledge; non-2xx retries.

## What this repo provides

Following the `whmcs-secrets-from-kv` pattern — Key Vault is the single source of truth, GitHub
consumes the key at runtime via OIDC, nothing secret is committed.

| Piece                                              | Purpose                                                                                                                                          |
| -------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------ |
| `.github/actions/zeffy-secrets-from-kv`            | OIDC → KV → exports masked `ZEFFY_API_KEY`.                                                                                                      |
| `scripts/zeffy-api-common.ps1`                     | `Invoke-ZeffyApi` (Bearer + 429 backoff) + `Get-ZeffyList` (cursor pagination).                                                                  |
| `scripts/zeffy-payments-export.ps1`                | Read-only payments → CSV (filters + date window).                                                                                                |
| `scripts/zeffy-contacts-export.ps1`                | Read-only donors → CSV.                                                                                                                          |
| `scripts/zeffy-campaigns-export.ps1`               | Read-only campaigns/events → CSV.                                                                                                                |
| `.github/workflows/401-zeffy-campaigns-export.yml` | **401. Zeffy - Campaigns Export** — dispatch-only; campaigns only; artifact `zeffy_campaigns`. Inherently PII-free.                              |
| `.github/workflows/402-zeffy-payments-export.yml`  | **402. Zeffy - Payments Export (PII masked)** — dispatch-only; payments with donor PII **masked**; artifact `zeffy_payments` (public-repo safe). |
| `.github/workflows/403-zeffy-contacts-export.yml`  | **403. Zeffy - Contacts Export (PII masked)** — dispatch-only; donors with PII **masked**; artifact `zeffy_contacts` (public-repo safe).         |

There is **one workflow per endpoint** and **no PII workflow** — none of them write donor PII to an
artifact. All three are **dispatch-only** (no schedule) while we evaluate the API. Unlike WHMCS
(which needs APIM for a static egress IP), **Zeffy has no IP allowlist**, so the runner calls
`api.zeffy.com` directly with the Bearer key — no APIM hop required.

The `-IncludePii` switch still exists on the payments and contacts scripts for **local/private** use
(run the script yourself and route the output to a private destination); **no workflow passes it**.

## Privacy: which calls are PII-free

The repo is **public**, so the workflows never write donor PII to an artifact. The scripts default
to **masked** output (`-IncludePii` is an opt-in switch that **no workflow passes** — it exists only
for local/private runs). Field-level reality from the OpenAPI spec:

| Endpoint                | Personal data in the response?                                                                                          | Workflow output                      |
| ----------------------- | ----------------------------------------------------------------------------------------------------------------------- | ------------------------------------ |
| `GET /api/v1/campaigns` | **None** — forms/events only (title, target, volume, dates, URLs).                                                      | ✅ fully safe, nothing to mask       |
| `GET /api/v1/payments`  | **Yes** — `buyer` (email/name/company), `buyer_questions`, `items[].questions`, `tribute.honouree_name`, `receipt_url`. | ⚠️ exported with PII columns blanked |
| `GET /api/v1/contacts`  | **Yes, by definition** — the donor directory (email/name/phone/address).                                                | ⚠️ exported with PII columns blanked |

So the **only call that never retrieves PII is `/campaigns`** (workflow 401). `/payments` (45) and
`/contacts` (46) return PII in the payload, but the masked export **writes the non-PII subset only**
(financials, status, campaign id, the pseudonymous `contact_id`/`id` UUIDs, giving
totals/counts/dates, country) — the PII passes through the ephemeral runner and is never persisted.
The masked fields are:

- **payments:** `buyer_email`, `buyer_first`, `buyer_last`, `buyer_company`, `receipt_url`.
- **contacts:** `email`, `first_name`, `last_name`, `phone_number`, `city`, `state`, `postal_code`.

To retrieve PII, run the script yourself with `-IncludePii` and route the output to a **private**
destination — there is intentionally **no workflow** that does this on the public repo.

## How it helps FFC

- **Donation reporting & reconciliation.** Pull every Zeffy payment into CSV on a schedule and
  reconcile against WHMCS / accounting — closing the loop on the existing one-way WHMCS→Zeffy import
  (workflow 213). You can now verify what actually landed in Zeffy instead of importing blind.
- **Donor (contact) sync.** `total_contribution`, `donation_count`, first/last donation dates, and
  address per donor — feed a CRM, mailing list, or year-end donor reports without manual exports.
- **Campaign / event dashboards.** Target vs. current `volume`, status, and event occurrences for
  every donation form and event — a live fundraising dashboard in the Actions job summary or pushed
  to a sheet.
- **Tax-receipt & finance support.** `eligible_amount`, `receipt_url`, refunds, and disputes per
  payment support receipting and finance audits.
- **Recurring-giving insight.** The `recurring` block (interval, subscription, period) surfaces
  monthly-donor retention and churn.
- **Real-time automation (phase 2).** The `payment.completed` webhook lets FFC react to a donation
  the moment it happens (e.g. trigger onboarding, thank-you, or a CRM update) via a small receiver
  (Cloudflare Worker or an APIM inbound op) — see "Webhook" below.

## One-time setup

1. **Org owner generates the key:** Zeffy → Settings → Integrations → generate API key.
2. **Store in Key Vault** (both copies hold the same value):
   ```bash
   az keyvault secret set --vault-name kv-ffc-admin-prod-cbm --name wr-all-ffc-zeffy-api-key   --value '<key>'
   az keyvault secret set --vault-name kv-ffc-admin-prod-cbm --name read-all-ffc-zeffy-api-key --value '<key>'
   ```
3. **Federated credential** for the `zeffy-prod` environment on `ffc-admin-kv-writer` (see the
   action README for the exact `az ad app federated-credential create` command).
4. **Repository Variables** `WR_ALL_FFC_AZURE_KV_CLIENT_ID` / `WR_ALL_FFC_AZURE_TENANT_ID` already
   exist (shared with the WHMCS flows).

Until the key is set, the Zeffy export workflows (**44–46**) fail fast (the action throws on the
placeholder value).

## Webhook (phase 2, not yet built)

To consume `payment.completed` in real time, stand up a receiver and register its URL in Zeffy →
Settings → Integrations. Natural fit: a Cloudflare Worker (or an APIM inbound operation) that
validates a shared secret (stored in KV as `*-ffc-zeffy-webhook-secret`) and processes the payment.
Tracked as a follow-up.

## Verification

- **Dry/local:** `pwsh -File scripts/zeffy-payments-export.ps1 -ApiKey '<key>' -OutputFile out.csv`
  (a personal/test key) — confirms pagination + CSV shape. Read-only; safe.
- **Sandbox:**
  `curl -H 'Authorization: Bearer <key>' 'https://api.zeffy.com/api/v1/payments?limit=1'` — note
  `api.zeffy.com` sits behind a Cloudflare bot-challenge, so a CI runner (real client) works; ad-hoc
  tooling may be challenged. Never echo the key.
- **CI dispatch:** run **401. Zeffy - Campaigns Export** (or **45**/**46**) once the key is in KV.
